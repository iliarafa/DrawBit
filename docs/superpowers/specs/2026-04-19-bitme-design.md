# Bitme — iPad Pixel Art App

**Status:** Draft — awaiting user sign-off before implementation planning.
**Date:** 2026-04-19

## Purpose

A native iPadOS app for creating pixel art on fixed-size canvases. Minimal UI, local-only storage, sole focus on being a useful pixel-art creator.

Non-goals: phone layout, Mac Catalyst, iCloud sync, layers, animation, selection tools, brush pressure, dithering, palette systems, onboarding/tutorials.

## Platform & scope

- **Native iPadOS**, Swift 5.10+, SwiftUI, Swift Concurrency.
- **Minimum target: iPadOS 17** (enables SwiftData, `@Observable`, modern gesture APIs).
- **iPad-only.** No iPhone layout. No Mac Catalyst.
- **Both orientations** supported. Canvas auto-centers on rotation.
- **Offline-only.** No network calls, no accounts, no sync.

## Canvas sizes

Four fixed sizes, selected at piece creation. Not resizable after creation.

| Size    | Pixels  | Raw bytes |
|---------|---------|-----------|
| 16×16   | 256     | 1 KB      |
| 32×32   | 1,024   | 4 KB      |
| 64×64   | 4,096   | 16 KB     |
| 128×128 | 16,384  | 64 KB     |

## Screens

Two screens, navigated via `NavigationStack`.

### GalleryView (launch screen)

- Grid of saved pieces. Each cell shows the piece's thumbnail.
- "**+ New**" button in the header opens a size picker (16 / 32 / 64 / 128) that creates a piece and pushes to the editor.
- **Tap** a piece → push to editor for that piece.
- **Long-press** a piece → context menu with **Rename**, **Duplicate**, **Delete**.
  - Delete shows a destructive `.confirmationDialog` confirmation. No undo after delete.
  - Rename opens an inline text field.
  - Duplicate creates a new piece with copied pixels and appended " copy" in the name.
- Names are auto-generated by default (`"<size>×<size> · <short date>"`) and optional to replace.
- No name is shown under thumbnails by default (cleaner grid). Name is visible inside the long-press menu.

### EditorView

Layout, landscape-first (portrait is the same pattern, just taller canvas area):

```
┌────────────────────────────────────────────────────────────┐
│ ‹ Gallery        32 × 32                ← → [Share]        │  Top bar
├────────────────────────────────────────────────────────────┤
│                                                            │
│                                                            │
│              ┌──────────────┐                              │
│              │   canvas     │  (free-transform: pan,       │
│              │  (checkered  │   pinch-zoom, two-finger      │
│              │   for alpha) │   rotate)                     │
│              └──────────────┘                              │
│                                                            │
│                                         320%               │

│ [Pencil][Eraser][Fill][Pick] │ [swatch][swatch][swatch][+] │  Bottom bar
└────────────────────────────────────────────────────────────┘
```

**Top bar** (thin): Back to gallery · piece size · ← undo · → redo · Share.

**Canvas surface** (flex-1): canvas is a free-floating view with no shadow, no outer halo — flat. Transparent pixels are shown with the standard checkerboard.

**Bottom bar** (pinned): tools on the left, vertical divider, horizontally scrollable recent-colors strip on the right. Swatches are **rectangles** with rounded corners. The selected swatch has a **same-color outline** with 2 px offset, giving a "slightly magnified" read — no contrasting border.

No settings screen, no preferences, no account.

## Tools

- **Pencil** — paints the selected color into the tapped/dragged pixel(s).
- **Eraser** — sets the tapped/dragged pixel(s) to fully transparent (alpha exactly 0).
- **Fill bucket** — 4-connected flood fill. Equality check is exact RGBA match, so transparent regions fill separately from opaque ones.
- **Eyedropper** — picks the color under the tap and sets it as the current color. If the picked sample is fully transparent, the current color is left unchanged (eyedropper is a no-op in that case).
- **Undo / Redo** — up to 50 steps per editing session. Each committed stroke / fill / erase is one step. Not persisted across closing the piece.

Implicit: no line tool, no rectangle tool, no selection, no brush sizes. The pencil paints exactly one pixel per point along the interpolated stroke path.

## Color picking

- **Recent colors strip**: the bottom-bar horizontal strip holds the most recent ~16 distinct colors used, newest first, capped and de-duplicated. Tap a swatch to select it.
- **+ button** on the strip opens the system `ColorPicker` (SwiftUI) in a popover. Selecting a color there sets the current color and prepends it to the recent-colors list.
- All colors are sRGB, 8-bit RGBA.
- Recent colors are app-wide (not per-piece), stored in `AppSettings`.

## Canvas transform & gestures

**The canvas floats freely.** The view supports the Procreate-style two-finger gesture: simultaneous pan + pinch-zoom + rotate.

- **Scale bounds:** 0.25× to 40×.
- **Rotation:** unbounded (free 360°), snapping to 0° at a small threshold (e.g., within 4°) so straightening back up is easy.
- **Reset:** double-tap with two fingers resets transform to "fit canvas centered in the editor area."
- **Rotation is view-only.** The pixel grid itself is always axis-aligned. Saved and exported PNGs are upright regardless of view rotation. Rotation state is not persisted — reopening a piece shows it upright.

**Input mode — auto-detected:**

- If **no Apple Pencil is paired / available**: finger draws; two-finger gesture transforms the canvas.
- If **an Apple Pencil is available**: Pencil draws; single-finger drag pans the canvas; two-finger pinch/rotate transforms the canvas. This prevents palm marks.
- Mode switches live when a Pencil connects/disconnects during the session.

**Stroke cancellation:** if a second finger lands mid-stroke, the in-progress stroke is discarded (not committed to undo).

## Background / transparency

- **New canvas is fully transparent** (alpha = 0 everywhere). The standard checkerboard pattern is shown behind it to indicate transparency.
- If the user wants a solid background, they fill-bucket it.
- Exported PNG has a real alpha channel.

## Pixel grid overlay

- Faint grid lines automatically appear when zoom level ≥ 4×, disappear when below that.
- No toggle. Grid is always 1 device pixel wide regardless of zoom (hairline), drawn between pixel cells, so it never obscures the art.

## Export / share

- **Share** button in the top bar opens a sheet presenting integer scale options:
  - 16×16 canvas → default **16×** (256 px output).
  - 32×32 canvas → default **16×** (512 px).
  - 64×64 canvas → default **8×** (512 px).
  - 128×128 canvas → default **4×** (512 px).
  - User can pick 1× / 2× / 4× / 8× / 16× from a simple selector.
- Export renders the pixel data via nearest-neighbor into a new bitmap of the target size and hands it off via the iOS share sheet (`ShareLink` or `UIActivityViewController`).
- The bitmap context is created with **explicit sRGB color space** (`CGColorSpace(name: CGColorSpace.sRGB)`) — never `deviceRGB` — so exported files look identical across devices including iPad Pro's P3 display.
- PNG is written with an embedded sRGB profile.
- The current view rotation does **not** affect the export. Always upright.

## Data model & storage

Local SwiftData, no sync.

```
@Model Piece
  id: UUID
  name: String?              // nil = use generated default at display time
  size: CanvasSize           // enum {s16, s32, s64, s128}
  pixels: Data               // raw RGBA bytes, length = size*size*4
  thumbnail: Data            // PNG, regenerated on every save
  createdAt: Date
  updatedAt: Date

@Model AppSettings   // singleton
  recentColors: [String]     // hex strings, newest first, cap 16
  lastUsedTool: Tool         // for editor-open default
```

**Pixel format.** Flat `Data` buffer of raw RGBA bytes, straight alpha (not premultiplied). Rendered to screen by wrapping the bytes in a `CGDataProvider`, building a `CGImage` with `CGImageCreate`, then drawing through the SwiftUI `Canvas`.

**Save strategy.** After every committed action (stroke end, fill tap, erase end), pixels + thumbnail are regenerated and written on a background task. Writes are debounced at ~100 ms to coalesce rapid strokes. SwiftData handles the transaction.

**Undo/redo.** In-memory stack of pixel snapshots scoped to the current editing session, max depth 50. At 128×128 that's ~3 MB worst case — trivial. Cleared when leaving the editor. Explicit design choice: undo across piece-open/close is **not** supported.

**What's NOT in the data model (YAGNI):** layers, frames, palette objects, tags, folders, favorites, iCloud records.

## Fidelity & quality guarantees

These are the rules the implementation must uphold so pixel-art output is crisp and exports match what's on screen.

1. **Nearest-neighbor interpolation everywhere upscaled.** `CGInterpolationQuality.none` on all `CGContext` draws of the pixel image — in the editor (including during transform/rotate/zoom), in thumbnail generation, and in PNG export.
2. **No antialiasing anywhere in the render pipeline.** Explicitly set `shouldAntialias = false` on render contexts so even the canvas outer bounds render as hard edges.
3. **Bresenham's line interpolation** between consecutive touch samples within a stroke — every pixel along the drag path gets painted, even at flick speed.
4. **Coalesced touches.** Read `UIEvent.coalescedTouches` to capture the full touch-sample stream (up to 240 Hz with Apple Pencil). Combined with Bresenham, no gaps.
5. **First pixel on tap.** A tap with no drag paints that pixel immediately — do not wait for a drag event before committing the first point.
6. **Deterministic touch → pixel mapping.** After applying the inverse of the current view transform, pixel index = `floor(x / pixelSize)` / `floor(y / pixelSize)`. Same rule for every tool. Out-of-canvas touches are rejected before flooring.
7. **One color space end-to-end: sRGB, 8-bit RGBA, straight alpha.** Swatch color == stored pixel == exported PNG byte. Deliberate choice to NOT use Display P3, because pixel art is meant to be shared and sRGB is the safe interop target.
8. **Explicit sRGB in export `CGContext`.** Create the export bitmap with `CGColorSpace(name: CGColorSpace.sRGB)` — never `deviceRGB` — to avoid baking P3 values into exports on iPad Pro.
9. **True zero alpha on erase.** Eraser writes alpha = 0 exactly; other components preserved as they were. Ensures clean composite on any background in other apps.
10. **Render at device scale.** The editor's `CGImage` is drawn at the iPad's native @2x / @3x backing scale — no intermediate low-res surface in the view layer.
11. **Integer-scale export only.** 1× / 2× / 4× / 8× / 16×. No fractional scales.
12. **Pixel data is the source of truth.** Edits mutate the raw `Data`. Thumbnails, rendered CGImages, and exported PNGs are derived and disposable — we never round-trip through a rendered image as working state.

## Edge cases

- **Drawing + transform collision.** Second finger touching down mid-stroke cancels the stroke (no partial commit).
- **Fill on transparent.** Fill-bucket treats `(0,0,0,0)` as a color; filling a transparent region fills only that connected area.
- **Eyedropper on transparent.** No-op — current color unchanged. Do not switch to a "transparent color" that the pencil could then paint with.
- **Delete.** Destructive `.confirmationDialog`. Once deleted, gone.
- **Backgrounding mid-edit.** Worst case loses the currently in-progress stroke (not yet committed). All previously committed actions are already persisted.
- **Apple Pencil mid-session connect/disconnect.** Input mode switches live. No toggle exposed in UI.
- **Transform bounds.** Scale clamped to [0.25×, 40×]; rotation snaps to 0° within a 4° threshold. Double-tap (two fingers) resets to fit.
- **Very small exports.** A raw 16×16 export is unusably small; default scale picker starts at 16× for tiny canvases.

## Testing strategy

- **Unit tests** for all of `Drawing/`:
  - Pencil placement at exact grid coordinates.
  - Bresenham interpolation correctness on known test cases (diagonals, verticals, steep lines).
  - Flood fill: connected regions, transparency handling, bounds.
  - Eyedropper: correct sample, no-op on transparent.
- **Unit tests** for `Rendering/`:
  - `pixelsToCGImage` round-trip: known bytes → image → same bytes.
  - Thumbnail generation at the gallery thumbnail size produces expected output.
  - PNG export at 1× produces expected bytes; scaled exports produce integer-multiple sizes with nearest-neighbor fidelity (spot-check specific pixels).
- **Unit tests** for `PieceRepository` with an in-memory SwiftData container: create, save (including thumbnail regen), rename, duplicate, delete.
- **One UI test** (happy path) on iPad simulator: new piece → draw pixels → back to gallery → reopen → pixels present.
- **Manual device testing** required for: transform gestures, Apple Pencil / finger auto-detect, ProMotion smoothness, share-sheet integration. These do not automate reliably.

## Risks

1. **Gesture conflicts** between draw and two-finger transform. *Mitigation:* prototype the combined gesture in isolation early, validate on device before wiring full UI.
2. **Rendering performance** at 128×128 with grid overlay at 120 Hz on ProMotion iPads. *Mitigation:* confirm the `Data → CGImage` draw path hits frame budget on-device; fallback is caching the `CGImage` and invalidating only changed regions between strokes.
3. **SwiftData writes blocking the main thread** during thumbnail regen. *Mitigation:* thumbnail regen and persistence run on a background task; editor state does not await them.
4. **Stroke sampling gaps** on fast drags. *Mitigation:* coalesced touches + Bresenham (items 3–4 of Fidelity section).
