This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

The Xcode project is generated by [xcodegen](https://github.com/yonaskolb/XcodeGen) from `project.yml`. `DrawBit.xcodeproj/` is gitignored.

Prereqs: `brew install xcodegen` (Xcode 16+ already installed).

**Regenerate the project** whenever you add, rename, or move a `.swift` file — the project references source files by path and will go stale:

```bash
rm -rf DrawBit.xcodeproj && xcodegen generate
```

**Build** (simulator):

```bash
xcodebuild -project DrawBit.xcodeproj -scheme DrawBit \
  -destination 'platform=iOS Simulator,name=iPad Pro 13-inch (M5)' build
```

If the named simulator isn't installed, list available ones with `xcrun simctl list devices available | grep -i iPad` and substitute.

**Run all tests:**

```bash
xcodebuild -project DrawBit.xcodeproj -scheme DrawBit \
  -destination 'platform=iOS Simulator,name=iPad Pro 13-inch (M5)' test
```

**Run a single test class or method** (fastest feedback loop during TDD):

```bash
# whole class:
xcodebuild ... test -only-testing:DrawBitTests/FillTests
# single method:
xcodebuild ... test -only-testing:DrawBitTests/FillTests/testFillEmptyCanvasFillsAll
# UI tests (slower):
xcodebuild ... test -only-testing:DrawBitUITests
```

**Device compile-check** (no signing required):

```bash
xcodebuild -project DrawBit.xcodeproj -scheme DrawBit -configuration Release \
  -destination 'generic/platform=iOS' CODE_SIGNING_ALLOWED=NO build
```

## Architecture

The app splits into four layers with strict one-way dependencies — **UI depends on State depends on Persistence depends on Rendering/Drawing/Models**. Drawing and Rendering have no UI or SwiftData imports and are covered by fast unit tests.

```
Views (SwiftUI, UIKit input)
  └─> State (EditorState: @Observable, session-scoped, undo/redo)
        └─> Persistence (PieceRepository wraps ModelContext)
              └─> Rendering (CoreGraphics: CGImage, thumbnails, PNG)
                    └─> Drawing (pure logic: PixelGrid, tools, Bresenham)
                          └─> Models (SwiftData + value types)
```

Key boundaries to preserve when extending:

- **Pure-logic rule:** anything in `DrawBit/Drawing/` and `DrawBit/Rendering/` imports only `Foundation` / `CoreGraphics` / `ImageIO` / `UniformTypeIdentifiers`. No SwiftUI, no SwiftData, no UIKit. This is what makes them trivially testable. New drawing tools or export paths belong here, not inside views. **One narrow exception:** `Rendering/MP4Exporter.swift` also imports `AVFoundation` / `CoreVideo` / `CoreMedia` for H.264 time-lapse encoding — still no UIKit/SwiftUI/SwiftData, still unit-tested (it writes a temp `.mp4` and re-reads it with `AVAsset`).
- **`PieceRepository` is the only SwiftData seam.** Views call the repository; they do not touch `ModelContext` directly. If you need a new persistence operation, add a method on `PieceRepository` — don't inline SwiftData calls into a view.
- **`EditorState` is in-memory and session-scoped.** Undo/redo snapshots, current tool, current color, and view transform live here. It is cleared when the editor closes. Do not persist transform state to `Piece`.
- **Canvas transform is view-only.** `state.translation`, `state.scale`, and `state.rotation` never alter the stored pixel grid. Exported PNGs are always axis-aligned. A new feature must not "bake" transform into pixels.

## Non-obvious invariants

These are load-bearing; breaking them silently corrupts exports or kills perceived quality.

- **`PixelGrid.init(data:size:)` defensively copies its `Data` argument** (`self.data = Data(data)`). A sliced `Data` has a non-zero `startIndex`; without the copy, `data[0]` traps and `CGDataProvider` reads the wrong bytes into the CGImage. See the regression test `testSlicedDataIsNormalized`.
- **`pixelsToCGImage` uses an explicit `CGColorSpace(name: CGColorSpace.sRGB)`** — never `deviceRGB`. On iPad Pro's P3 display, `deviceRGB` would silently bake wide-gamut values into exported PNGs. The same applies to the `CGContext` created in `ThumbnailRenderer.render` and `PNGExporter.export`.
- **Nearest-neighbor everywhere:** every rendering code path sets `interpolationQuality = .none` and `setShouldAntialias(false)` on its `CGContext`, and UI `Image` views use `.interpolation(.none).antialiased(false)`. Pixel edges stay hard at any zoom.
- **Straight alpha, not premultiplied at the data level.** `PixelGrid` stores raw RGBA bytes. `CGImage` is built with `.premultipliedLast`, which is identical to straight alpha as long as alpha is only 0 or 255 — so the eraser must write `alpha = 0` exactly (`RGBA.transparent`), and drawing tools must never produce partial alpha. If that changes, the render path needs revisiting.
- **Stroke drawing uses `UIEvent.coalescedTouches` + Bresenham line interpolation.** A stroke without Bresenham leaves gaps on fast drags. New drawing tools that respond to `onStrokePoint` get this for free because `CanvasInputView.touchesMoved` already walks coalesced samples and interpolates.
- **The tracing reference photo is display-only — never a `Layer`.** `Piece.referenceImageData` (a size-capped sRGB JPEG via `ReferenceImageProcessor`) renders only as a dimmed backdrop in `CanvasView`; it never enters `Frame.layers`, `Compositor`, or any exporter. That is exactly what lets a *partially transparent* photo appear on screen without violating the alpha-0/255 + "topmost-opaque-wins" layer invariants above. Don't turn it into a real layer or composite it into exports.
- **The time-lapse recorder snapshots the composite, not the tools.** `EditorState.recordKeyframe()` runs at each commit point (`commitStroke` / `commitStructuralChange` / `commitSequenceChange`) and stores a zlib-compressed `Compositor.composite` of the **active frame's layers only** — never the reference photo. Frames dedupe + halve to a cap (`TimelapseRecording`, `Timelapse.maxStoredKeyframes`), persist in the additive optional `Piece.timelapseData` column (lightweight migration, no V2 schema — see below), and export via `MP4Exporter` as an H.264 clip on an **opaque matte** (`MP4Exporter.defaultMatte`; MP4 has no alpha). Undo/redo do **not** record (the replay stays forward-moving); `EditorView` calls `recordKeyframe()` once more on close/share to guarantee the clip ends on the real canvas. The recording is display/export-only — keep it out of `frameData`, `Frame.layers`, and every non-`MP4Exporter` export path.
- **SwiftData has a single live-typed schema; do not add a second `VersionedSchema`.** `DrawBitSchemaV1.models` lists the live `Piece`/`AppSettings`, so a `DrawBitSchemaV2` listing the same live types would compute an identical checksum and CoreData aborts with "Duplicate version checksums detected" at on-disk container creation — i.e. a launch crash that unit/in-memory tests don't catch. Additive optional/defaulted fields (like the reference-photo columns) ride **inferred lightweight migration** with the one schema; `ReferenceUpgradeMigrationTests` proves a real pre-field store upgrades cleanly. A genuine *reshaping* migration must first freeze V1 into nested snapshot model types. If the store still can't open, `DrawBitApp` catches the failure and `StoreRecovery.moveStoreAside` renames it aside (never deletes) so the app launches fresh instead of boot-looping.

## Test conventions

- Unit tests mirror the source tree: `DrawBit/Drawing/Fill.swift` → `DrawBitTests/Drawing/FillTests.swift`. Keep that mirror when adding tests.
- Unit tests `@testable import DrawBit` — internal (non-`public`) symbols are visible. Default access control is fine.
- SwiftData tests use an in-memory container: `ModelConfiguration(isStoredInMemoryOnly: true)`. See `PieceRepositoryTests` for the setup pattern; reuse it for any new persistence test.
- UI tests launch the app with `launchArguments = ["-UITest-reset"]`, which is checked in `DrawBitApp.swift` to switch the `ModelContainer` to `inMemory: true`. Tests get a clean store every run.
- **UI test waits use a generous `waitForExistence(timeout: 15)`.** The full UI suite runs ~15 tests back-to-back on one simulator; under that contention, transitions/re-renders are slower than in isolation, so tight 2–3s waits flake. `waitForExistence` returns the instant the element appears, so a large timeout costs nothing on success — only on a genuine failure. Keep *negative* assertions (`XCTAssertFalse(...waitForExistence(timeout: 1))`) and optional-UI probes short on purpose.
- **Animations are disabled under UI test, and that is what makes the suite stable.** `UITestSupport.isRunning` (in `RootView.swift`) is true whenever any `-UITest*` launch arg is present. `DrawBitApp.init` then calls `UIView.setAnimationsEnabled(false)` (kills UIKit-backed animations: navigation push/pop, sheet & popover presentation, the **text edit menu**, keyboard, `List` row moves), and every SwiftUI `.animation(_:value:)` / `withAnimation` site is gated to `nil` under test (landing "PRESS START" blink, `LayerRow` pulse, `LayersPanel` slide+opacity). **Why it matters:** a running animation means XCUITest never sees the app reach "idle" (its automatic post-event sync point), and an element mid-transition can satisfy `waitForExistence` while not yet being stably hittable. The historical flake was `LayersPanelUITests.testOpenPanelAndRenameLayer`: under load, tapping "Select All" on the *animated* edit menu missed, so the rename produced a mangled name. When adding UI, gate any new `.animation`/`withAnimation`/`.repeatForever` on `!UITestSupport.isRunning` — especially `.repeatForever`, which otherwise wedges XCUITest's idle wait permanently.
- **Reads that depend on async view rebuilds must poll, not read once.** Inserting a `FrameRow`/`LayerRow` triggers a SwiftUI rebuild plus thumbnail render that is *not* animation-gated, so a `.count`/`.exists` read taken immediately after a `.tap()` can race it. Poll (`for _ in 0..<50 where count < N { Thread.sleep(0.1); count = … }`) as the frame/layer tests do.
